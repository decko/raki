# Documentation Milestone

**Date**: 2026-04-19
**Status**: Ready
**Version**: No version bump — docs only

## Goal

Get an external team from "interested" to "running RAKI against their sessions" in under 10 minutes. All deliverables are documentation and examples — no `src/` changes.

## Context

RAKI v0.4.0 is feature-complete for Phase 1+2 (330 tests, 48 PRs, 5 milestones). There are interested external teams but no second adopter yet. Current docs are a 30-line README and `--help` text. The adoption blockers are:

1. No getting-started path — users don't know how to begin
2. No example data — can't try the tool without bringing their own sessions
3. No adapter guide — teams with different session formats don't know how to integrate
4. No results interpretation — metrics are numbers without meaning
5. No ground truth guidance — bad ground truth makes Ragas metrics meaningless

## Deliverables

### 1. Example Session Data

`examples/sessions/` — 5 session-schema sessions + 1 Alcove session. All use a fictional "web-api" project (no references to internal projects).

**Session diversity** (each exercises a distinct code path):

| Session | Scenario | Key features |
|---|---|---|
| `pass-clean/` | All phases succeed, no rework | Baseline happy path |
| `rework-cycle/` | Verify fails → review → rework → pass | Generational files (implement.json.1, .2), rework_feedback_injected event |
| `critical-findings/` | Review finds critical issues | review.json with severity:critical findings, multi-reviewer |
| `partial-failure/` | Implement phase fails, no verify | Incomplete session, missing later phase files |
| `multi-reviewer/` | Parallel specialist reviews merged | Two reviewer_started/completed pairs, review_merged event |
| `alcove-session.json` | Alcove format equivalent | Single-file transcript with tool calls |

Each session directory includes a `_README.md` (3-5 lines) explaining what the session demonstrates. This file is not parsed by adapters (they ignore `.md` files).

The `rework-cycle/` session's `events.jsonl` covers the full event taxonomy: `phase_started`, `phase_completed`, `phase_failed`, `rework_feedback_injected`, `reviewer_started`, `reviewer_completed`, `review_merged`.

### 2. Example Ground Truth

`examples/ground-truth/curated.yaml` — 5 entries demonstrating the full taxonomy:

```yaml
- id: gt-001
  question: "How should API input validation be implemented?"
  expected_contexts:
    - "Use Pydantic models for request validation"
    - "Return 422 with field-level errors"
  reference_answer: "API endpoints validate input using Pydantic models..."
  domains: [api, validation]
  difficulty: easy
  knowledge_type: procedure
  expected_phase: implement

- id: gt-002
  question: "What are the rate limiting constraints?"
  # ... (hard difficulty, constraint type, triage phase)
```

Entries cover: easy/medium/hard difficulty, all 4 knowledge types (fact, procedure, constraint, context-dependent), different expected phases.

### 3. Example Manifests

`examples/` — 3 annotated YAML manifests with inline comments:

| File | Purpose |
|---|---|
| `raki-minimal.yaml` | Bare minimum — just sessions path |
| `raki-full.yaml` | All options: sessions + ground truth + filters |
| `raki-alcove.yaml` | Alcove format sessions |

Inline YAML comments explain each field. No separate manifest documentation — the comments are the docs.

### 4. Example Reports

`examples/reports/` — pre-generated JSON + HTML from the example data.

- Generated by running `raki run --manifest examples/raki-minimal.yaml --no-llm --include-sessions`
- HTML report committed for browsing
- Screenshot of HTML report added to README (`docs/images/report-screenshot.png`)

### 5. Getting Started Guide

`docs/getting-started.md` — under 100 lines. Structure:

1. **Prerequisites** — Python 3.12+, uv
2. **Install** — `uv pip install raki` or clone + `uv sync`
3. **Quick Start** — run against bundled examples (copy-pasteable commands, 60 seconds to output)
4. **Prepare Your Data** — what session data looks like, link to schema reference and examples
5. **Understanding the Output** — what the CLI summary means, how to open the HTML report
6. **Next Steps** — link to interpretation guide, adapter guide, ground truth guide

Includes verbatim terminal output with callouts annotating key lines.

### 6. Results Interpretation Reference

`docs/interpreting-results.md` — what each metric means and what to do about it.

For each metric:
- What it measures (one sentence)
- Valid range and direction (higher/lower is better)
- Thresholds: green/yellow/red guidance
- Common patterns and what they indicate
- Example: "knowledge_miss_rate 1.0 means 100% of rework-triggering findings had no related knowledge in context — your knowledge base has gaps for these topics"

Includes a "Common Patterns" section:
- High miss rate + low severity = retrieval gaps on easy questions
- Low verify rate + high rework = systemic quality issues
- High cost + low rework = expensive but correct sessions

### 7. Ground Truth Curation Guide

`docs/curating-ground-truth.md` — how to write effective ground truth entries.

Covers:
- What makes a good question (discriminating, not obvious)
- How to write expected_contexts (specific enough to match, not exact strings)
- Difficulty calibration (easy = answered by any knowledge entry, hard = requires multi-entry synthesis)
- Knowledge type selection (fact vs procedure vs constraint vs context-dependent)
- Domain assignment (use controlled vocabulary, avoid synonyms)
- Anti-patterns: too broad, too obvious, paraphrased contexts, duplicate coverage

### 8. Adapter Authoring Guide

`docs/writing-adapters.md` — under 150 lines.

Structure:
1. **Overview** — what adapters do, the two built-in examples
2. **The Protocol** — `detect()` and `load()` signatures with type annotations
3. **Detection** — must be cheap (file stat, header read), never parse the full file
4. **Loading** — parse session data into `EvalSample`, handle missing fields
5. **Redaction** — call `redact_sensitive()` before populating `EvalSample`
6. **Registration** — add to registry in `cli.py`
7. **Testing** — fixture patterns, edge cases to cover
8. **Complete Example** — a minimal working adapter (30-40 lines)

## What's NOT in scope

- No `src/` changes
- No version bump
- No new features (raki report --diff, raki generate, etc.)
- No CI integration examples (deferred to Phase 3)
- No improvement playbook ("how to fix low scores") — deferred until interpretation guide exists
- No CLI reference doc — `--help` with enriched epilog examples is sufficient

## Success Criteria

- A developer who has never seen RAKI can go from `git clone` to a working evaluation report in under 10 minutes by following the Getting Started guide
- An external team with a different session format can write a working adapter by following the adapter guide
- A user can look at a RAKI report and understand what actions to take by reading the interpretation reference
