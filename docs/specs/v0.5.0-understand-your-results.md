# v0.5.0 — Understand Your Results

**Date**: 2026-04-19
**Status**: Ready

## Goal

Make RAKI's output worth sharing. Three features that transform raw numbers into actionable insights: re-render reports from JSON, redesigned HTML with plain-English explanations, and diff between two runs.

## Scope

3 items, ordered by dependency:

1. **`raki report`** — re-render CLI/HTML from existing JSON results
2. **#68 HTML report overhaul** — score cards + drill-down redesign
3. **`raki report --diff`** — compare two evaluation runs

## 1. `raki report` — Re-render from JSON

### Command

```
raki report --input results/eval-abc123.json [flags]

  --input, -i     Path to JSON report (required)
  --html          Output HTML file path (default: same dir as input, .html extension)
  --output, -o    Output directory (overrides --html default)
```

### Behavior

- Loads a saved `EvalReport` from JSON via `load_json_report()`
- Renders CLI summary to terminal
- Generates HTML report to disk
- If `sample_results` is empty (stripped by default), renders aggregate scores only and shows warning: "Per-session drill-down unavailable — original report was generated without --include-sessions"
- HTML report shows empty-state message in drill-down section, not a blank space

### Implementation

- Add `report` command to `cli.py`
- Reuse existing `print_summary()` and `write_html_report()` — they already accept `EvalReport`
- No new modules needed, just CLI wiring

## 2. HTML Report Overhaul (#68)

### Score Cards — Redesign

**Summary sentence** above cards:
> **91% of sessions passed on first try**, with an average of 0.4 rework cycles. Reviewers found 3 critical, 8 major, and 25 minor issues across all sessions. Average cost: **$7.42/session**.

The summary is generated from aggregate scores — plain English, key numbers bolded.

**Card layout:**
- Hero card for Verify Rate (2-column span, larger value)
- Standard cards for Rework Cycles and Cost/Session
- Full-width severity distribution bar replacing the unintuitive 0.58 score

**Every card has three layers:**

```
VERIFY RATE                              ← metric name (links to interpreting-results.md)
How often the agent's work passes        ← plain-English subtitle (always visible)
all checks on the first try

91%                                      ← value (large, colored)
████████████████████░░                   ← progress bar
[higher is better]  Target: >85%         ← direction badge + threshold
```

**Plain-English subtitles (always visible):**

| Metric | Subtitle |
|---|---|
| Verify Rate | How often the agent's work passes all checks on the first try |
| Rework Cycles | How many times the agent had to redo its work after feedback |
| Cost / Session | How much each agent task costs in API fees |
| Review Findings | How many issues reviewers found, broken down by severity |
| Knowledge Miss Rate | How often rework happened because the agent lacked the right reference material |
| Context Precision | How much of what the retriever pulled in was actually relevant |
| Context Recall | How much of the needed information the retriever successfully found |
| Faithfulness | How closely the agent's output sticks to the facts in its source material |

**Direction badges + thresholds:**

| Metric | Direction | Threshold |
|---|---|---|
| Verify Rate | higher is better | Target: >85% |
| Rework Cycles | lower is better | Good: <1.0 |
| Cost / Session | (no direction) | Shows range: min – max |
| Severity | (distribution bar) | Traffic-light label: Clean / Moderate / Severe |
| Knowledge Miss Rate | lower is better | Target: <0.20 |
| Context Precision | higher is better | Target: >0.80 |
| Context Recall | higher is better | Target: >0.80 |
| Faithfulness | higher is better | (no threshold, experimental) |

**Severity distribution bar** replaces the single score:

```
┌──────────────────────────────────────────┐
│ ███ 3   ████████ 8   ██████████████ 25   │
│ crit     major        minor              │
└──────────────────────────────────────────┘
Legend: ● 3 critical  ● 8 major  ● 25 minor    [Moderate]
```

**Conditional display:**
- Knowledge Miss Rate: hidden when all sessions have `knowledge_context=None`. Footnote: "Knowledge Miss Rate omitted — no retrieval context available in sessions."
- Retrieval Quality section: hidden entirely in `--no-llm` mode. Footnote: "Retrieval metrics omitted — run without --no-llm to enable LLM-backed evaluation."
- Metric names link to corresponding section in `docs/interpreting-results.md` on GitHub

**Color thresholds for Rework Cycles:**
- Green: <1.0
- Yellow: 1.0–2.0
- Red: >2.0

### Per-Session Drill-Down — Redesign

**Verdict-based rows** (reviewed by UXD + skeptical QE):

```
▶ FAIL    115 — implement failed · [2 critical]              $4.20  0:42
▶ REWORK   53 — 2 cycles · [1 critical] [2 major]           $26.10  7:34
▶ PASS    102 — 5 phases · [1 minor]                          $8.50  4:12
▶ PASS    103 — 5 phases                                      $6.80  3:45
```

Each row shows:
- Verdict badge (PASS / REWORK / FAIL) — leftmost, high-contrast, colored
- Session ID
- Detail text — "2 cycles", "implement failed", "5 phases"
- Severity badges on REWORK/FAIL rows only — critical/major pills
- Cost + duration right-aligned
- Left border colored by verdict (red / yellow / none)

**Sort order:** FAIL → REWORK → PASS, cost descending within each group.

**"Worst Sessions" section** above full drill-down — shows only FAIL + REWORK sessions with a count badge: "2 sessions need attention".

**Expanded view** (when clicking a row):
- Phase timeline with status dots (completed / failed) and duration
- Review findings with severity badges, file location, reviewer name

### Empty States

Every conditional section has an explicit empty-state message:
- No findings: "No review findings in this session"
- No phases: "Phase data unavailable"
- No retrieval metrics: "Retrieval metrics require LLM judge — run without --no-llm to enable"
- No drill-down data: "Per-session details unavailable — regenerate with --include-sessions"

## 3. `raki report --diff` — Compare Two Runs

### Command

```
raki report --diff baseline.json compare.json [flags]

  --output, -o    Output directory for diff report
  --html          Output HTML diff file path
  --json          Print diff summary as JSON to stdout
```

### Behavior

File-based comparison. No `parent_run_id` or model changes — takes two JSON report files and compares them.

### Session Matching

Match sessions by `session_id`. Partition into:
- **Matched** — same session_id in both reports
- **New** — in compare but not in baseline
- **Dropped** — in baseline but not in compare

Report coverage prominently: "42/58 matched, 16 new, 4 dropped"

### Output Structure

**Warning banner** (if session counts differ):
> ⚠ 4 sessions dropped, 16 new — aggregate deltas based on 42 matched sessions only

**Aggregate deltas table:**

| Metric | Baseline | Compare | Delta | Sig? |
|---|---|---|---|---|
| Verify Rate | 78% | 91% | +13% | ✓ |
| Rework Cycles | 1.2 | 0.4 | -0.8 | ✓ |
| Cost / Session | $12.30 | $7.42 | -$4.88 | |

Delta column color-coded: green for improvements, red for regressions (respects `higher_is_better`).

Significance flag: simple annotation per metric. For binary metrics (verify rate), use binomial proportion confidence interval. If delta < margin of error at n sessions, show "~" (within noise) instead of "✓". Full statistical testing deferred to future work.

**Changed sessions** (grouped by transition):

```
REWORK → PASS (5 sessions)               ← improvements first
▶ PASS  102 — was REWORK (1 cycle)
▶ PASS  115 — was REWORK (2 cycles)

PASS → REWORK (2 sessions)               ← regressions prominent
▶ REWORK 131 — was PASS · [1 critical]
▶ REWORK 133 — was PASS · [2 major]

New sessions (16)                         ← collapsed by default
Dropped sessions (4)                      ← collapsed, with warning
```

Regressions (PASS → REWORK, PASS → FAIL) sorted before improvements. Within each group, sorted by severity.

**CLI summary** for `--diff`:

```
$ raki report --diff baseline.json compare.json

Comparing eval-abc123 → eval-def456
Matched: 42/58 sessions (16 new, 4 dropped)

  Verify Rate       78% → 91%  (+13%)  ✓ significant
  Rework Cycles     1.2 → 0.4  (-0.8)  ✓ significant
  Cost / Session  $12.30 → $7.42  (-$4.88)

Improvements: 5 sessions (REWORK → PASS)
Regressions:  2 sessions (PASS → REWORK)

Diff report written:
  HTML → results/diff-abc123-vs-def456.html
```

### What Requires `sample_results`

The diff command needs per-session data to compute session-level transitions. Both input reports must have been generated with `--include-sessions`. If either report has empty `sample_results`:
- Aggregate deltas still computed (from `aggregate_scores`)
- Changed sessions section shows: "Per-session comparison unavailable — both reports must be generated with --include-sessions"

## Data Model

No model changes. The existing `EvalReport` schema is sufficient:
- `run_id` — identifies each report (already exists)
- `aggregate_scores` — for delta computation (already exists)
- `sample_results` — for session-level transitions (already exists, populated since v0.3.0)
- `manifest_hash` — for comparing evaluation configs (already exists)

`parent_run_id` deferred to the replay harness milestone. File-based diff is sufficient for v0.5.0.

## Specialist Review Summary

3 rounds of specialist reviews informed this design:

**Score cards** (UXD + RAG + Data Viz):
- Plain-English subtitles always visible (no tooltips — broken on mobile/Slack)
- Severity as distribution bar, not single score
- Hide Knowledge Miss Rate when structurally absent
- Hide Retrieval Quality section in --no-llm mode
- Direction badges + thresholds on every metric
- Metric names link to docs (not modals — YAGNI)

**Drill-down** (UXD + skeptical QE):
- Option B: verdict badges, FAIL-first sort
- Severity badges on REWORK/FAIL only
- Worst Sessions section above drill-down

**Diff output** (PM + UXD + RAG + QE, unanimous):
- Option B: aggregate deltas + changed sessions only
- Dropped sessions as first-class warning banner
- Significance flag per metric (lightweight)
- Regressions before improvements

**Run lineage** (RAG + Architecture):
- File-based diff, no parent_run_id — "the real risk is locking in the wrong lineage model"
